# Structural performance findings (current UI) and design rules for the new code

Input to the migration: structural performance issues in the existing viewer that
the new architecture (`02-architecture.md`) must design against. Findings, not
patches - the point is the design rule each one yields.

## Method

Hypotheses were generated by four targeted code sweeps (render granularity,
per-frame allocation/re-derivation, DOM churn/forced layout, data-model memory
shape), then verified empirically against the running viewer (dev-server on
:3001, headless Chromium via Playwright + CDP): CPU profiles per interaction
storm, `Performance.getMetrics` deltas (layout counts, heap), the viewer's own
`?prof=1` per-panel render log, and reload cycles. Two datasets: the demo trace
(x1: 10.57 MB raw, 294k events, 27.5k spans, 2 workers) and the same trace
repeated 8x via repeatable `trace=` params (x8: 84.6 MB raw, ~2.4M events) -
just under the 100 MB open cap.

Verdicts: **CONFIRMED** (measured), **MECHANISM** (code-proven at `file:line`;
not triggerable with the demo data shape), **NEGATIVE** (hypothesis tested and
rejected).

## Measured baseline

| Metric | x1 (10.57 MB raw) | x8 (84.6 MB raw) |
|---|---|---|
| Load wall time | 3.8 s | 12.2 s |

| One full `renderAll` | 36.9 ms (lanes 20.2) | 859.7 ms (lanes 843.5) |

| Heap churn during one pan storm | +18 MB | +11 MB |
| Reload-cycle heap (3x) | 118.7 -> 118.2 -> 118.2 MB | 810.1 -> 810 -> 810 MB |

## Findings

### F1. Lane rendering is not pixel-bounded: stroked primitives scale with event count - CONFIRMED

The codebase's own perf discipline (pixel downsampling + run-length-coalesced
`fillRect`, `trace_analysis.js:337`, `:280-305`) applies to FILLS only. Stroked
primitives bypass it. Measured: in the x8 pan storm, `stroke()` self-time was
**89.0 s of 117.2 s sampled (76%)**, with `setLineDash` another 4.8 s;
`renderAll` grew 36.9 ms -> 859.7 ms (23x) for 8x events at identical pixel
size, with the lanes phase 20.2 -> 843.5 ms with only 2 lanes.

Sources (mechanism): the per-lane local-queue step line builds a path with one
vertex per visible sample - per-sample, not per-column (`viewer.html:3135-3180`);
open-ended poll markers draw dashed edges with per-item `setLineDash` state
churn (`viewer.html:2985-2999`); boot/gap dashes similar. GC visible in the same
profile (489 ms) from per-frame churn.

**Design rule (amends N1):** every lane/panel primitive - strokes included -
must be pixel-bounded and batched: downsample series to one vertex (or run) per
pixel column before pathing, group same-style strokes into one path, hoist dash
state changes. "Pixel-bounded" is the contract for ALL draw primitives, not just
fillRect.

### F2. Monolithic renderAll with no dirty tracking; most triggers bypass the coalescer - CONFIRMED

`renderAll` (`viewer.html:2579`) unconditionally re-renders every lane and every
panel. Of ~35 call sites, only three route through the RAF coalescer
(`scheduleRenderAll`, `viewer.html:2654`); wheel zoom (measured: 6 ticks -> 6
synchronous full renders, 1:1), span-filter keystrokes (`viewer.html:2177-2182`),
legend-chip toggles (`:2151`, `:2058`), panel fold (`:1416-1419`), TZ/format
toggles (`:4919-4931`), window resize (`:6488`, un-debounced - the sibling
heatmap page debounces, `index.html:1432-1443`), and selection clicks
(`:5288-5293`) all call it directly. Even the correctly-coalesced waker-label
hover (`viewer.html:6433-6436`, change-gated and RAF-scheduled) still runs a
FULL all-panel renderAll for an effect consumed only by lane highlight - with no
dirty tracking, the coalescer can only cap frequency, never scope. At x8 every
full render costs ~860 ms of main-thread stall.

**Design rule (validates 05 sections 2.2/2.3):** all state mutations go through
the store; rendering subscribes per slice and the scheduler coalesces to one RAF
tick; the renderer registry maps changed slices to affected canvases only.
Direct render calls from event handlers are prohibited - the new code must have
NO path from input event to render that skips the scheduler.

### F3. Forced synchronous layout on every pointer event in the tooltip/overlay path - CONFIRMED

Measured: 171 layout passes for 183 mousemoves, both scales. Mechanism: all four
tooltip paths write `tooltip.innerHTML` then immediately read
`tooltip.offsetWidth/offsetHeight` in `placeTooltip` (`viewer.html:1446-1453`;
call sites `:6400->6409`, `:3664->3666`, `:4081->4083`, `:4171->4173`). The
lanes scroll handler calls `renderCrosshair()` synchronously per scroll event
(`viewer.html:4963-4965`), which writes `#info-panel.innerHTML`
(`:4771`) then reads `getBoundingClientRect` (`:4790-4793`) - a forced reflow
per scroll tick. `renderCrosshair` also reallocates the full-screen overlay
canvas backing store on every invocation even when its size is unchanged
(`viewer.html:4779-4782`); same pattern per lane in `renderLane`
(`:2801-2806`).

**Design rule:** the overlay/transient channel performs zero DOM writes; the
tooltip is a declarative component whose placement uses cached dimensions (or
transform-based positioning) - never a measure-after-write; scroll is
RAF-coalesced like every other input; canvas backing stores resize only on
geometry change. Layout reads are batched before writes inside
`lib/canvas/layout.ts` (single read pass per frame).

### F4. Parsed-trace heap is ~10-13x the raw trace bytes - CONFIRMED (with a NEGATIVE sub-result)

Measured: 139 MB heap for 10.57 MB raw (x1), 864 MB for 84.6 MB raw (x8). At the
100 MB open cap this implies ~1 GB of JS heap. Mechanisms (code-proven, in the
frozen core): `events[]` is a polymorphic array of 6+ hidden classes -
PollStart/PollEnd even swap the insertion order of the
`localQueue`/`globalQueue` pair (`trace_parser.js:612-613` vs `:627-628`),
enough for distinct hidden classes - defeating monomorphic access in
every hot loop; each poll is materialized ~3x (event objects + per-worker refs +
poll-span objects, `trace_analysis.js:484`, `:520-568`); event objects carry ~7
permanently-zero fields (~56 dead bytes/event, `trace_parser.js:623-634`); the
decoder churns 2 throwaway objects + a BigInt + a decimal string per event
(`decode.js:309-326`; `trace_parser.js:555`); CPU-sample callchains become
un-interned per-sample hex-string arrays (`trace_parser.js:717-719`).

NEGATIVE: no reload leak - heap is stable across 3 reload cycles at both scales;
module-level caches in the core are bounded and hold no trace references
(`trace_analysis.js:228`).

**Design rule + constraint tension:** the core is frozen (01-technical-constraints.md H2), so this
inflation ships into the new UI unchanged. Accept and budget it: the
architecture assumes ~10x-raw heap at the cap (documented as an N-budget), the
store holds references to core output and never copies per-event data, and all
derived views are windowed/pixel-bounded so they never re-materialize the event
set. Columnarizing the core (struct-of-arrays, interning) is real leverage but
is a separate post-migration phase with its own ADR and test plan - not part of
this migration.

### F5. Per-frame re-derivation of frame-invariant data - CONFIRMED (representative case) / MECHANISM (catalog)

Measured: `selectSpanRenderSet` + `computeSpanLayout` = 472 ms of the x1 pan
storm profile (~7.5 ms per frame of pure re-derivation at 27.5k spans); the
`?prof=1` log shows `spanPanel(allSpans=27524)` ~10 ms on every renderAll frame.
Mechanism catalog of the same pattern: `renderTaskDetail` re-collects and
re-sorts every poll of every worker per frame while a task is selected
(`viewer.html:4203-4209`); `renderLane` recomputes per-trace-invariant
`workerGaps` per lane per frame plus fresh closures and coalescer objects
(`:2815-2817`, `:2949-2958`); visibility scans are full O(N) despite sorted
arrays and an existing `lowerBound` helper (`renderSpanPanel`
`:3239-3243`, custom events `:3394-3398`); the heatmap page recomputes
`bootTransitions` (with an internal sorted copy) per row per redraw while its
sibling caches `row.tiled`/`row.gaps` sit one line away (`index.html:1196` vs
`:1082-1083`); `computeSpanLayout` allocates a string Map key per render-set span per frame
(`trace_analysis.js:1363`). The codebase demonstrably knows the fix - color
caches, `getSpanDurations` cache, `taskForEvent` WeakMap - it is just applied
inconsistently.

**Design rule:** derived data is cached in the store keyed by the slices it
depends on (trace, selection, viewport-window), invalidated by slice change, and
computed via binary-search windows over the sorted invariants (encoded in the
types). Per-frame work touches only per-frame inputs.

### F6. Interaction handlers do content work inline - MECHANISM

The lane tooltip performs two linear scans of `allSpans` per raw mousemove, the
second with no early exit (`viewer.html:6329-6343`, `:6374-6380` - `continue`
not `break` over a start-sorted array); `updateInfoPanel` walks
`activeTaskSamples` linearly from index 0 per crosshair frame
(`viewer.html:4760-4765`). Not measurable on the span-light demo (hover stayed
flat at both scales); scales with span-heavy traces per the code's own
multi-million-span comments.

**Design rule (validates 05 section 2.5):** interaction modules translate events
into store actions and overlay updates only; every content query they need goes
through indexed, binary-search-backed helpers in `lib/trace/query.ts`.

### F7. Chrome rebuilds its subtree and listener set wholesale - MECHANISM

`renderRawTable` rebuilds the whole tbody and attaches one listener per checkbox
per search, then re-queries the DOM for every selection count
(`index.html:1535-1590`); legend chips rebuild the entire chip set and re-bind
every listener per toggle, then trigger a full renderAll
(`viewer.html:2144-2155`, `:2052-2062`); heatmap labels are rebuilt then
`plot.clientWidth` is read, forcing a reflow per browse render
(`index.html:1054-1106`); `setHeatmapSelection` toggles a class across every
label row on each selection change (`index.html:1418-1422`).

**Design rule (validates 05 section 2.4):** declarative templated components
with keyed diffing and event delegation at the component root; no per-row
listener rebinding; component renders never interleave layout reads.

## What this changes in 02-architecture

1. N1 extended: pixel-bounding applies to every draw primitive including
   stroked paths; series are downsampled to per-column vertices before pathing;
   same-style strokes batch into one path (F1).
2. New NFR: heap budget ~10x raw trace bytes at the 100 MB cap (~1 GB); new
   code adds no per-event copies; derived views are windowed (F4).
3. New NFR: no render path may bypass the store scheduler; input handlers never
   call render functions (F2, F6).
4. N17 sharpened: overlay channel does zero DOM writes; tooltip placement never
   reads layout after a write; canvas backing stores resize only on geometry
   change (F3).
5. Store design: derived-data caches keyed by dependency slices are a
   first-class store feature, not per-component improvisation (F5).
6. Post-migration backlog item (own ADR): columnarize the frozen core's event
   storage (struct-of-arrays, string interning, monomorphic shapes) - measured
   justification here, out of scope for the migration itself (F4).
7. Design opportunity for `lib/trace/load.ts`: parse/analysis currently runs on
   the main thread (the 3.8 s / 12.2 s load walls are main-thread time,
   mitigated only by streaming progress yields). The frozen core is
   environment-agnostic JS (it already runs under Node), so the new load
   pipeline can execute it in a Web Worker WITHOUT touching the core - the
   worker posts the parsed result to the store. Evaluate at load-pipeline
   design time; not a core change.
